Pipeline — Giải thích toàn bộ flow

Tài liệu này giải thích từng bước của pipeline từ raw CSV files đến dashboards, bao gồm lý do tại sao mỗi quyết định kỹ thuật được đưa ra theo cách đó.

0 Pipeline là gì và tại sao cần nó?

Hiểu bức tranh tổng thể trước khi đi vào chi tiết.

Hãy tưởng tượng MoMo xử lý hàng triệu giao dịch mỗi ngày. Dữ liệu thô (CSV files, JSON) không thể dùng trực tiếp để phân tích hay phát hiện gian lận — nó cần được làm sạch, làm giàu, tổng hợp, và lưu trữ theo cách tối ưu cho từng use case.

Pipeline là chuỗi các bước xử lý data có thứ tự, tự động, và có thể chạy lại mà không tạo ra kết quả sai.

Tại sao chia thành nhiều layers?
Mỗi layer có mục đích riêng và "hợp đồng" riêng (schema, quality guarantees). Nếu có lỗi ở staging, raw layer vẫn còn nguyên vẹn để reprocess. Đây gọi là defense in depth — phòng thủ nhiều lớp.

2 Tech Stack — Tại sao chọn từng tool?

Mỗi tool giải quyết một vấn đề cụ thể, không phải chọn cho có.

🐘 HDFS (Hadoop)

Distributed file system. Lưu Parquet files theo dạng phân tán. Tại sao không dùng local disk? Vì HDFS scale ngang — thêm node là có thêm storage và bandwidth.

⚡ Apache Spark

Distributed processing engine. Đọc Parquet từ HDFS, xử lý song song trên nhiều CPU/node. Tại sao không pandas? Pandas không scale — 10GB file là chết ngay.

🐝 Apache Hive

Table catalog (Metastore). Lưu metadata: table X có những columns nào, kiểu gì, nằm ở HDFS path nào. Spark và Trino đều đọc catalog này.

🔷 Trino

MPP SQL query engine. Query Parquet trên HDFS nhanh 10-100x so với Hive vì không dùng MapReduce. Superset và dbt connect vào đây.

🌿 dbt

SQL-based transformation framework. Mỗi model là 1 SQL file, có lineage tracking, tự động test, có documentation. Chuẩn industry cho warehouse layer.

🔴 Apache Airflow

Orchestrator. Scheduler chạy pipeline đúng thứ tự, đúng giờ, xử lý dependency, retry khi fail, gửi alert. DAG = Directed Acyclic Graph = sơ đồ luồng công việc.

📊 Apache Superset

BI/visualization tool. Kết nối vào Trino, tạo charts và dashboards mà không cần viết code. Business team dùng cái này.

🐳 Docker Compose

Containerization. Toàn bộ stack chạy trong containers — không phụ thuộc vào máy cụ thể. docker compose up là có ngay môi trường đầy đủ.

3 Layer 0 — Raw Source Files

Điểm xuất phát — data chưa được xử lý gì.

Raw files nằm tại data/raw/. Đây là "sự thật gốc" — nếu có vấn đề gì trong pipeline, luôn có thể quay về đây để reprocess.

Vấn đề của raw data

  • CSV không có type info — tất cả đều là string, kể cả số
  • Amount có thể là "$1,234.56", "1.234,56", hoặc "1,234" (ambiguous)
  • Date format có thể không nhất quán
  • Có null values, duplicates, invalid records
  • Không có relationship enforcement — card_id trong transactions có thể không tồn tại trong cards
Tại sao không sửa thẳng vào raw files?
Raw layer là immutable (bất biến). Không bao giờ sửa, xóa, hay overwrite raw data. Lý do: nếu logic transform của bạn sai, bạn vẫn có thể chạy lại từ đầu. Đây là nguyên tắc quan trọng nhất trong data engineering.

4 Layer 1 — Ingestion (RAW HDFS Layer)

Đọc source files và lưu vào HDFS dưới dạng Parquet. Không transform gì cả.

Ingester làm gì?

Ingester = "bơm nước" — đọc data từ nguồn và đẩy vào HDFS. Chỉ làm 3 việc:

  1. Đọc CSV/JSON với explicit schema (không để Spark tự infer)
  2. Thêm metadata columns: _ingested_at, _source_file, _batch_id
  3. Write ra HDFS dưới dạng Parquet partitioned

Tại sao dùng explicit schema?

Cách sai — inferSchema
# BAD: Spark phải đọc toàn bộ file 1 lần để infer type
# Nếu cột amount có "N/A" → Spark infer là String → downstream fail
df = spark.read.option("inferSchema", "true").csv(path)
Cách đúng — explicit schema
# GOOD: Define schema trước, đọc với type đã biết
# Nếu source gửi sai type → fail sớm, rõ ràng, dễ debug
schema = StructType([
    StructField("id",     StringType(), nullable=True),
    StructField("date",   StringType(), nullable=True),  # String trước, parse sau
    StructField("amount", StringType(), nullable=True),  # String trước, clean sau
    # ...
])
df = spark.read.schema(schema).option("header", "true").csv(path)

Pattern: Template Method trong BaseIngester

Đây là design pattern quan trọng. BaseIngester define flow cố định, mỗi ingester chỉ implement phần khác nhau:

# BaseIngester — flow không thể thay đổi
def run(self):
    df = self.read_source()           # ← subclass implement
    df = self._add_metadata(df)       # ← base class xử lý (không thể bỏ qua)
    df = self.add_partition_cols(df)  # ← subclass implement
    self._write(df, output_path)      # ← base class xử lý (không thể bỏ qua)

# TransactionIngester — chỉ implement phần của nó
class TransactionIngester(BaseIngester):
    def read_source(self):
        return spark.read.schema(RAW_CSV_SCHEMA).csv(path)

    def add_partition_cols(self, df):
        return df \
            .withColumn("year",  F.year(F.col("date"))) \
            .withColumn("month", F.month(F.col("date"))) \
            .withColumn("day",   F.dayofmonth(F.col("date")))
Tại sao Template Method pattern?
Đảm bảo mọi ingester đều thêm metadata columns và implement idempotency check. Không thể "quên" các bước quan trọng vì chúng nằm trong base class. Nếu thêm CardIngester sau này, chỉ cần implement 3 methods, flow tự động đúng.

Output: HDFS Parquet với partitions

hdfs://namenode:9000/data/lake/raw/transactions/
  year=2023/month=01/day=01/part-00000.parquet   (128MB)
  year=2023/month=01/day=02/part-00000.parquet
  year=2023/month=02/day=01/part-00000.parquet
  ...

Tại sao Parquet thay vì giữ nguyên CSV? 3 lý do:

  • Columnar format — query cột amount không cần đọc các cột khác
  • Compression — Snappy compress thường 3-5x nhỏ hơn CSV
  • Built-in statistics — mỗi column group có min/max → predicate pushdown

5 Layer 2 — Staging (Clean + Enrich)

Spark đọc từ raw, làm sạch, làm giàu, validate. Kết quả vẫn là Parquet trên HDFS.

Staging làm những việc này theo thứ tự:

1. Schema enforcement & type casting

# Cast từ String sang đúng type
df = df \
  .withColumn("mcc",              F.col("mcc").cast(IntegerType())) \
  .withColumn("transaction_date", F.to_timestamp(F.col("date"), "yyyy-MM-dd HH:mm:ss"))

2. Amount cleaning — vấn đề phức tạp nhất

Amount có thể đến ở nhiều format tùy source system và locale:

Raw valueCó thể làKết quả
"$1,234.56"US format rõ ràng1234.56 PARSE OK
"1.234,56"EU format rõ ràng1234.56 PARSE OK
"₫200,000"VND, phẩy=thousand200000.0 PARSE OK
"1,234"1234 (US) hoặc 1.234 (EU)?NULL AMBIGUOUS
"1.234"1234 (EU) hoặc 1.234 (decimal)?NULL AMBIGUOUS
"(500.00)"Accounting negative-500.0 PARSE OK
Nguyên tắc quan trọng nhất
Không bao giờ tự động sửa khi ambiguous. Không bao giờ drop record. Sai 1 giao dịch trong financial system = compliance issue. Record ambiguous → quarantine table → human review.

Thay vì 1 cột amount, staging có 5 cột:

amount_raw        -- giữ nguyên raw string, không bao giờ xóa
amount            -- parsed value, NULL nếu không parse được
amount_currency   -- "USD", "VND", ... nếu detect được
amount_format     -- "US" | "EU" | "CLEAN" | "VN" | "AMBIGUOUS" | "INVALID"
amount_parse_note -- lý do nếu ambiguous: "could be 1234 or 1.234"

3. Deduplication

# transaction_id là unique key
# Nếu ingest lại cùng file → không tạo duplicate
df = df.dropDuplicates(["transaction_id"])

4. Enrichment — join với dimension tables

# Join với mcc_codes để thêm mcc_description
# broadcast() vì mcc_codes nhỏ (~300 rows) → không shuffle
df = transactions_df.join(
    F.broadcast(mcc_df),
    on="mcc",
    how="left"
)

# Join với cards để thêm card_brand, card_type
df = df.join(
    F.broadcast(cards_df.select("card_id", "card_brand", "card_type")),
    on="card_id",
    how="left"
)

# Attach fraud label
df = df.join(
    fraud_labels_df,
    on="transaction_id",
    how="left"
)
Tại sao dùng broadcast() cho mcc và cards?
Sort-merge join (default) cần shuffle cả 2 bảng qua network = chậm. Broadcast join: copy bảng nhỏ lên TẤT CẢ executors = không shuffle gì cả. Với mcc (~300 rows), broadcast nhanh hơn 10-50x.

Transaction Tracing — tìm giao dịch từ user_id

-- Có user_id, tìm toàn bộ context giao dịch
SELECT
    t.transaction_id, t.amount, t.transaction_date,
    u.full_name, u.phone_number,
    c.card_brand, c.card_number_masked,
    m.merchant_name, m.mcc_description,
    f.is_fraud, f.risk_score, f.txn_count_last_24h
FROM warehouse.fact_transactions t
JOIN warehouse.dim_users u     ON t.user_id     = u.user_id
JOIN warehouse.dim_cards c     ON t.card_id     = c.card_id
JOIN warehouse.dim_merchants m ON t.merchant_id = m.merchant_id
LEFT JOIN warehouse.feat_fraud_features f
                               ON t.transaction_id = f.transaction_id
WHERE t.user_id = 'USR_12345'
  AND t.year = 2023   -- partition pruning: chỉ đọc year=2023
  AND t.month = 6    -- partition pruning: chỉ đọc month=6
ORDER BY t.transaction_date DESC;

8 Layer 5 — Orchestration (Airflow)

Airflow đảm bảo pipeline chạy đúng thứ tự, đúng giờ, tự động retry khi fail.

DAG là gì?

DAG = Directed Acyclic Graph = đồ thị có hướng không có chu trình. Mỗi node là 1 task, mỗi edge là dependency (task A phải xong trước task B).

ingest_transactions ──┐
ingest_users        ──┤
ingest_cards        ──┤──► validate_raw ──► spark_staging ──► hive_repair
ingest_mcc          ──┘                                          │
                                                                 ▼
                                                           dbt_warehouse
                                                                 │
                                                                 ▼
                                                           dbt_test ──► notify

Tại sao cần Airflow? Không cần chỉ dùng cron?

  • Dependency management — cron không biết task A cần xong trước task B
  • Retry — task fail tự retry N lần với exponential backoff
  • Backfill — chạy lại pipeline cho ngày quá khứ (khi fix bug)
  • Monitoring — xem task nào đang chạy, task nào fail, mất bao lâu
  • Alert — gửi Slack/email khi pipeline fail

10 Spark Optimization — Tại sao quan trọng?

3 thứ tốn tiền nhất trong Spark và cách giải quyết.

1. Shuffle — bottleneck #1

Shuffle xảy ra khi Spark cần redistribute data giữa executors qua network. groupBy, join (không broadcast), repartition đều gây shuffle.

Gây shuffle — tránh
# Sort-merge join: shuffle CẢ HAI bảng qua network
transactions_df.join(mcc_df, on="mcc")
Không shuffle — dùng broadcast
# Broadcast join: copy bảng nhỏ lên mỗi executor → không shuffle
transactions_df.join(F.broadcast(mcc_df), on="mcc")

2. Small files — giết HDFS NameNode

NameNode lưu metadata của MỖI file trong RAM. 1 triệu files nhỏ → NameNode OOM → cả cluster chết.

# Target: mỗi file ~128MB (= 1 HDFS block)
# Trước khi write, repartition để đúng số files
df.repartition("year", "month", "day") \
  .write.partitionBy("year", "month", "day") \
  .parquet(output_path)

Compaction job chạy mỗi đêm để merge files nhỏ lại.

3. Data skew — straggler problem

Merchant "Grab" có 10 triệu transactions. Executor xử lý "Grab" chạy 10000x lâu hơn. Toàn bộ job wait executor đó.

# AQE tự detect và xử lý skew
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")

Adaptive Query Execution (AQE) — Spark 3.0+

AQE là gì và tại sao luôn bật?
Trước Spark 3.0, query plan được lập từ đầu dựa trên ước tính. AQE cho phép Spark re-optimize ở runtime dựa trên actual statistics: tự coalesce partitions nhỏ, tự convert sang broadcast join, tự split skewed partition. Bật nó là được optimization miễn phí. 
spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")

11 Storage Lifecycle — Hot / Warm / Cold

Data cũ không cần giữ ở cùng tier với data mới. Tiered storage giảm chi phí drastically.

TierThời gianStorageCompressionFile size
HOT 0–7 ngày HDFS SSD Snappy (fast) 128MB
WARM 8–90 ngày HDFS HDD ZSTD (better ratio) 256MB
COLD 91+ ngày HDFS HDD ZSTD max 256MB (aggregated only)
Snappy vs ZSTD?
Snappy: compress/decompress rất nhanh, tỷ lệ nén trung bình. Dùng cho hot data — đọc nhiều, cần latency thấp.
ZSTD: compress chậm hơn nhưng tỷ lệ nén tốt hơn ~20%. Dùng cho warm/cold — đọc ít, ưu tiên storage cost.

12 Amount Cleaning — Case Study

Ví dụ điển hình về việc "cái tưởng đơn giản lại phức tạp nhất".

Logic detect format

def detect_format(raw: str):
    has_dot   = '.' in raw
    has_comma = ',' in raw

    # Có cả 2: nhìn cái nào ở CUỐI → đó là decimal separator
    if has_dot and has_comma:
        if raw.rfind('.') > raw.rfind(','):
            return "US"    # "1,234.56" → chấm sau phẩy → chấm là decimal
        else:
            return "EU"    # "1.234,56" → phẩy sau chấm → phẩy là decimal

    # Chỉ có 1 loại separator, và phần sau separator có đúng 3 chữ số:
    # → KHÔNG THỂ biết đây là thousand separator hay decimal
    if (has_dot or has_comma) and len(raw.split('.')[-1]) == 3:
        if currency == 'VND':   # context giúp resolve
            return "VN"
        return "AMBIGUOUS"    # → quarantine, không tự sửa

Quarantine flow

Record ambiguous/invalid
    │
    ▼
staging.transactions_quarantine
    │
    ├──► Rule-based resolution (tự động nếu có đủ context)
    │       ↓ resolved → back to staging
    │
    └──► Manual review queue
            ↓ human confirm
            ↓ resolved → back to staging
